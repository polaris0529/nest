# PRD(제품 요구사항 정의서) 가이드 정리

> 참고: 구글 PM Omar Eduardo Fernández의 WHY 기반 PRD 작성법 및 일반적 PRD 구성 요소 정리

---

## 1. PRD란?

**PRD(Product Requirements Document)** = 제품을 만들거나 업데이트할 때 **요구사항을 개괄적으로 설명하는 문서**.

- 기획: 비전·전략 정의  
- 디자인: 사용자 요구 충족 여부 확인  
- 개발: 기능 목적·구현 확인  
- 테스트: 기술 요구사항 충족 확인  
- 출시: 시장/사용자 요구 충족 확인  

**목적:** 개발팀·이해관계자가 요구사항을 공유하고, 같은 기준으로 개발에 착수할 수 있게 하는 것.

---

## 2. PRD 작성 방법 (6가지 원칙)

### ① 사용자 문제 + 비즈니스 우선순위 근거를 넣기
- **왜 이 기능을 지금 개발하는지** 명확히 한다.
- 사용자 조사, 옵션 평가, 기술 한계, UX, 리소스·예산, 일정 등을 PRD에 담고, **이해관계자·의사결정권자와 합의**한다.
- “꼭 지금 해결해야 하는 일인가?”를 검토해 불필요한 기능 개발을 줄인다.

### ② 해결 방안의 세부 정보 추가
- 제안한 **해결 방안의 구체적 내용**을 적는다.
- 사용자 조사 문서, 회의 내용, 기술 한계, UX 고려, 리소스·타임라인, **다른 팀/기술에 대한 종속성**, 관련 링크 포함.
- 협업 팀이 PRD만 보고 이해할 수 있게 쓴다.

### ③ UX·개발 설계 문서를 참조
- PRD는 **무엇을, 왜** 구축하는지 정의하고, **어떻게** 구축하는지는 UX·개발 설계 문서로 분리한다.
- 설계 단계에서 옵션에 대한 피드백을 반영하고, **우선순위를 다시 점검**한다.
- PM은 사용자 가치와 복잡도의 균형을 설명할 수 있어야 한다.

### ④ PRD를 업데이트해 기능 범위 확정
- **범위 결정**과 **중요한 절충안**을 PRD에 반영한다.
- 기술적 고려, 타팀 종속성, 개인정보·법적 이슈, “특정 버전으로 먼저 구축하기로 한 이유” 등을 기록한다.
- UX·개발 설계 문서와 **중복을 피하고**, PRD는 결정과 근거에 집중한다.

### ⑤ 자주 공유하고 의견 받기
- PRD와 필수 변경 사항을 **자주 공유**하고, 이해관계자와 **피드백**을 받는다.
- 결정 내용과 **근거**를 빠르게 확인하고, 장단점을 함께 검토한다.

### ⑥ 개발 중에도 PRD 계속 업데이트
- 개발 중에는 **범위 vs 일정** 절충이 생긴다. 범위를 줄여서라도 중요한 부분을 먼저 출시하는 경우가 많다.
- PRD를 **지속 업데이트**하고, 일정·범위 변경을 팀에 공유해 계획을 유연하게 조정한다.

## 3. PRD 구성 요소 (7가지)

> 회사·프로젝트에 따라 생략·추가 가능. “해야 할 일이 명확하고 합의가 되어 있다”가 중요하다.

| # | 구성 요소 | 내용 |
|---|-----------|------|
| 1 | **제품 개요(요약·배경)** | 제품 목적, 해결할 문제, **왜 지금 해결해야 하는지 근거**. 설문·인터뷰·지표·VOC 키워드·시장/경쟁 상황 등 데이터로 보강. |
| 2 | **비즈니스 목표·지표** | 개발 이유, 기대 성과, **달성할 비즈니스 결과**. 비즈니스 목적 분석 → KPI 식별 → 목표 수치 설정. 현재 vs 예상 데이터 비교. |
| 3 | **핵심 고객·사용자 정의** | 제품을 **누가** 쓰는지 정의. 기능이 많으면 사용자 세분화. “이 제품으로 갈증이 해소되는 사용자”가 겪는 문제를 명확히 기술. |
| 4 | **핵심 사용자 여정(CUJ) / 사용자 스토리** | 정의한 사용자가 **목표를 달성할 수 있는지** 시나리오로 작성. **요구에 집중**하고, 특정 해결책에만 매달리지 않기. 문장 또는 사용자 여정 지도로 정리. |
| 5 | **기능적 요구사항** | 제품이 **반드시 수행해야 할 기능**과 **우선순위**. 개발자와 소통하며 작성. Must / Should / Could Have 구분. |
| 6 | **프로젝트 일정·배포 계획** | 개발·출시 일정. PM 단독이 아니라 **개발팀·이해관계자·의사결정권자**와 협의. 마케팅·영업·CS 등 출시에 맞춘 준비 가능하도록 공유. |
| 7 | **기타 참고 문서** | 와이어프레임, 스토리보드, UX 리서치, 트렌드·레퍼런스 등. 설득력과 해결 방향 검토에 활용. |

## 4. 기능 요구사항 우선순위 (MoSCoW)

| 구분 | 의미 | 예 (음식 주문 앱) |
|------|------|-------------------|
| **Must Have** | 없으면 제품을 쓸 수 없음 | 메뉴 선택, 주문·결제 |
| **Should Have** | 꼭은 아니지만 사용자가 매우 원함 | 메뉴 검색 |
| **Could Have** | 있으면 좋은 선택 기능 | 할인 정보 확인 |

---

## 5. 사용자 스토리 문장 형식

- **“(사용자 유형)은 (목적)를 위해 (필요/욕구)를 원한다.”**
- 예: “쇼핑을 자주 하는 나는, 결제 정보를 저장해 매번 다시 입력하지 않고, 빠르게 결제할 수 있기를 원한다.”

→ 사용자 중심으로 기능을 설계·개발하는 데 활용.

---

## 6. PRD 작성 시 유의사항

- 잘못된 PRD → **일정 지연, 잘못된 소통, 타겟 요구 미충족**으로 이어질 수 있음.
- **목적·요구사항·일정**을 명확히 하면서, **관련 이해관계자의 피드백을 반영**하는 것이 중요함.
- 팀/팀원, 진행 상태, 투자 가치, 정책, Q&A 등은 필요에 따라 추가 가능.

## 7. 이 문서와 `PRD.md`의 관계

- **이 파일(`PRD-가이드-정리.md`)**: PRD가 무엇인지, 어떻게 쓰는지에 대한 **작성 가이드**.
- **`docs/PRD.md`**: 실제 프로젝트용 **PRD 명세 템플릿** (개요, 사용자, 요구사항, 일정 등을 채워 넣는 문서).

프로젝트 PRD를 쓸 때는 이 가이드를 참고한 뒤 `PRD.md`에 내용을 채우면 된다.
